Relationship Model\n\n```\nusers 1───* accounts          (NextAuth OAuth links)\nusers 1───* sessions          (NextAuth DB sessions — optional)\nusers 1───* images            (uploaded files)\nusers 1───* prompts           (custom prompt templates)\nusers 1───* analysis_runs     (batch analysis jobs)\n\nprompts 1───* prompt_versions     (audit trail of edits)\nprompts 1───* analysis_runs — central identity record; mirrors what NextAuth expects (`id`, `email`, `name`, `image`, `email_verified`).\n- **`accounts`** — NextAuth's per-provider link (Google, GitHub, etc.), used when non-credentials sign-in is added.\n- **`sessions`** — present for NextAuth's database-session strategy; with the JWT strategy (US-024 default) it stays empty but available for a future toggle.\n- **`verification_tokens`** — NextAuth's email-verification flow, optional for the POC.\n\n### 3.2 Analysis Configuration\n\n- **`criteria`** — a lookup table seeded with the **7 criteria** explicitly named in the project idea (attractiveness, color_balance, selling_idea, trending_viral_concept, marketing_message, product_presentation, value_message). Modeling them as rows (instead of an enum) lets you add new criteria, edit descriptions, or hide them without a schema migration.\n- **`prompts`** — stores both the **system default** (`is_system_default = TRUE`) and **user-saved custom prompts**. The default row carries the well-engineered prompt written in US-009.\n- **`prompt_versions`** — append-only history so a user can re-run an analysis with an older version of their prompt (auditability, A/B testing).\n\n### 3.3 Image Storage\n\n- **`images`** — one row per uploaded file. The actual binary lives outside Postgres (filesystem, S3, R2…) referenced by `storage_path`. The row also stores the `sha256_hash` for **deduplication and cache lookups** (an explicit pipeline described in the project elaboration). `width`/`height` come from the Pillow preprocessing step (US-006). `metadata` is a `JSONB` for EXIF and future enrichment.\n\n### 3.4 Analysis Pipeline\n\n- **`analysis_runs`** — one row per user-initiated batch analysis. Carries the **prompt mode** (`default`, `custom`, `hybrid`), the resolved `prompt_id` (if any), the verbatim `custom_prompt_text` (if `use_default_prompt = FALSE`), the `llm_provider` / `llm_model`, and the **overall winner** (`winner_image_id`) plus free-form `overall_summary`. Token counts and `latency_ms` enable cost dashboards. A `CHECK` constraint enforces the invariant that exactly one of `use_default_prompt` ↔ `custom_prompt_text` is meaningful.\n- **`analysis_results`** — one row per image per run. Contains the human-readable fields (`summary`, `recommendation`, `strongest_criterion`, `weakest_criterion`, `confidence`) and the **full raw LLM response** in `JSONB` for replay/audit.\n- **`criterion_scores`** — narrow table holding the **0-10 score per criterion per result**. Having one row per (result × criterion) makes \"winner per criterion\" queries simple (used by the comparison grid in US-022) and keeps updates to scoring scales a metadata change, not a schema change.\n\n---\n\n## 4. 🏷️ Naming Conventions\n\n| Layer | Convention | Example |\n|---|---|---|\n| Tables | **plural**, `snake_case` | `analysis_runs`, `criterion_scores` |\n| Columns | `snake_case` | `winner_image_id`, `created_at` |\n| Primary key | `id` (UUID) | `id UUID PRIMARY KEY DEFAULT gen_random_uuid()` |\n| Foreign key | `<referenced_table_singular>_id` | `user_id`, `image_id`, `result_id` |\n| Timestamps | `created_at`, `updated_at`, `*_at` | `completed_at`, `email_verified` |\n| Booleans | `is_*`, `use_*` | `is_active`, `use_default_prompt` |\n| Enums (PG types) | `PascalCase` | `analysis_status`, `prompt_mode` |\n| Indexes | `idx_<table>_<col_or_purpose>` | `idx_analysis_runs_user_created` |\n| Triggers | `trg_<table>_<purpose>` | `trg_users_updated_at` |\n| Constraints | `<table>_<purpose>` | `criterion_scores_range` |\n\n---\n\n## 5. ⚡ Indexing Strategy\n\n| Index | Purpose |\n|---|---|\n| `users(email)` UNIQUE | NextAuth lookup; sign-in |\n| `accounts(provider, provider_account_id)` UNIQUE | NextAuth OAuth link |\n| `sessions(session_token)` UNIQUE + `sessions(expires)` | Session lookup + cleanup |\n| `images(user_id)`, `images(user_id, sha256_hash)` | Per-user gallery + dedup lookups |\n| `images(created_at DESC)` | Chronological sorting in galleries |\n| `prompts(is_system_default) WHERE is_system_default = TRUE` | Partial index — single hot lookup |\n| `analysis_runs(user_id, created_at DESC)` | History dashboards (US-031) |\n| `analysis_runs(status)` | Queue/worker polling |\n| `analysis_results(run_id)`, `analysis_results(image_id)` | Run detail page + re-analysis lookups |\n| `criterion_scores(result_id)`, `criterion_scores(criterion_id)` | Comparison grid aggregations |\n| `idx_prompt_versions_prompt_id` | Version history fetch |\n\nPartial indexes and composite indexes are preferred over blanket indexes to keep write amplification low while still covering the read patterns of the comparison dashboard and history views.\n\n---\n\n## 6. ⚖️ Trade-offs & Design Decisions\n\n| Decision | Choice | Rationale |\n|---|---|---|\n| **PK type** | `UUID` (`gen_random_uuid()`) | Avoids enumeration attacks on `/api/images/{id}` (US-007), allows client-side ID generation, plays well with distributed deploys. |\n| **Criteria** | Lookup table (not enum) | Lets you add/edit criteria without DDL migrations (REQ-04 evolution). |\n| **Prompt mode** | Enum + nullable FK + free text | Captures all US-009/US-011/US-012 states cleanly with a single `CHECK` invariant. |\n| **Image bytes** | Out-of-DB (filesystem/S3) + path in row | Postgres TOAST would bloat the DB; S3/R2 is the natural home for media. |\n| **Raw LLM response** | Stored in `JSONB` | Auditable, replayable, lets future versions re-score without re-calling the API. |\n| **Winner** | Stored pointer + recomputable | Stored for fast \"winner banner\" render (US-023); recomputable via aggregation over `criterion_scores` when criteria evolve. |\n| **Cascade rules** | `CASCADE` for owned data, `SET NULL` for cross-references, `RESTRICT` for lookup tables | Deleting a user wipes their data; deleting an image nulls the winner reference; deleting a `criteria` row is blocked if used. |\n| **Sessions table** | Included but unused in JWT mode | Future-proofs a single-line NextAuth config swap to DB sessions. |\n| **No `tags`, `comments`, `collections`** | Omitted | Not required by the 5 core requirements; can be added without breaking existing tables. |\n| **No `comparisons` table** | Omitted | A/B comparison mode (project elaboration, \"future\") can be modeled later by adding a `comparison_id` column on `analysis_runs`. |\n\n### What's intentionally NOT modeled\n- **Real-time progress stream** (SSE/WebSocket) — handled by in-memory pub/sub in the FastAPI process; no DB row needed.\n- **Token-usage billing ledger** — `analysis_runs.total_input_tokens` is enough for the POC; full ledger can be derived.\n- **Audit log table** — `prompt_versions`, `raw_response JSONB`, and `updated_at` triggers cover current audit needs.\n\n---\n\n## 7. 🧱 DDL\n\n```sql\n-- ============================================================\n-- PostgreSQL Schema: AI Image Analysis App\n-- Aligned with: FastAPI + Next.js + TailwindCSS + NextAuth\n-- Source: REQ-01..REQ-05, US-001..US-032\n-- ============================================================\n\n-- Enable required extensions\nCREATE EXTENSION IF NOT EXISTS \"pgcrypto\";
